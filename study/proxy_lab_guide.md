# Proxy Lab 가이드

Tiny를 완성했다면 다음 단계는 프록시 서버다.  
프록시는 브라우저와 웹 서버 사이에 서서 요청과 응답을 중계한다.

```text
Browser -> Proxy -> Web Server
Browser <- Proxy <- Web Server
```

## 프록시 서버를 왜 배우는가

- Tiny는 "서버가 요청을 받는 법"을 보여준다.
- Proxy는 거기에 더해 "서버이면서 동시에 클라이언트가 되는 법"을 보여준다.
- 따라서 소켓 흐름을 더 입체적으로 이해하게 된다.

## 과제 목표 정리

1. 순차적 요청 처리
2. 병렬적 요청 처리
3. 요청 캐시

## 먼저 풀어야 할 핵심 질문

1. 브라우저가 프록시에 보내는 요청 형식은 일반 서버 요청과 어떻게 다른가?
2. 프록시는 원격 웹 서버로 어떤 요청을 다시 만들어 보내야 하는가?
3. 응답을 읽는 동안 일부를 캐시에 저장하려면 버퍼를 어떻게 다뤄야 하는가?
4. 동시에 여러 클라이언트를 처리하면 어떤 데이터가 공유되는가?

## 1단계: 순차적 프록시

### 최소 처리 흐름

```text
Open_listenfd
  -> Accept
  -> 요청라인 읽기
  -> URI에서 host/port/path 파싱
  -> Open_clientfd로 원격 서버 연결
  -> 새 HTTP 요청 작성
  -> 원격 서버에 전달
  -> 응답을 읽어 브라우저에 전달
  -> 종료
```

### 중요한 포인트

- 프록시는 브라우저에게는 서버다.
- 동시에 원격 웹 서버에게는 클라이언트다.
- 따라서 `accept`와 `connect`를 한 프로그램 안에서 모두 사용한다.

### 주의할 헤더

Lab에서는 보통 아래 헤더를 명시적으로 다루게 된다.

- `Host`
- `User-Agent`
- `Connection: close`
- `Proxy-Connection: close`

핵심은:

- 브라우저가 준 요청을 그대로 전달하는 것이 아니라
- 프록시 규칙에 맞는 새 요청으로 정리해서 보내는 것이다.

## 2단계: 병렬 처리

### 왜 필요한가

순차 프록시는 한 요청을 처리하는 동안 다른 요청을 기다리게 만든다.

### 선택지

- 프로세스 기반
- 스레드 기반
- I/O 멀티플렉싱

이 과제에서는 보통 `thread-per-connection` 방식이 구현 난이도와 학습 효과가 적절하다.

### 초보자 관점에서의 권장 접근

1. 순차 버전을 완전히 동작시킨다.
2. 연결마다 스레드를 하나 생성한다.
3. 공유 데이터가 생기기 전까지는 락 없이 진행한다.
4. 캐시를 넣는 순간부터 동기화 전략을 추가한다.

## 3단계: 캐시

### 왜 필요한가

- 같은 요청이 반복되면 원격 서버에 다시 가지 않아도 된다.
- 응답 속도가 빨라지고 네트워크 사용량이 줄어든다.

### 기본 아이디어

- 요청 URI를 키로 사용
- 응답 바이트를 값으로 저장
- 너무 큰 객체는 저장하지 않음

### 제공된 상수

- `MAX_CACHE_SIZE`
- `MAX_OBJECT_SIZE`

이 상수는 "전체 캐시 한도"와 "개별 객체 한도"를 의미한다.

### 설계 시 질문

1. 캐시에 저장할 단위는 URI인가, Host+Path 조합인가?
2. 응답을 읽는 도중 어느 시점에 캐시에 넣을 것인가?
3. 캐시가 가득 차면 어떤 정책으로 내보낼 것인가? (`LRU` 권장)
4. 동시 요청 중 캐시에 대한 읽기/쓰기를 어떻게 보호할 것인가?

## Tiny와 Proxy의 차이

| 항목 | Tiny | Proxy |
| --- | --- | --- |
| 브라우저 요청 수신 | 예 | 예 |
| 원격 서버 연결 | 아니오 | 예 |
| 정적 파일 직접 제공 | 예 | 아니오 |
| 동적 CGI 실행 | 예 | 아니오 |
| 요청 재작성 | 거의 없음 | 중요 |
| 캐시 | 없음 | 과제 핵심 |
| 동시성 | 반복형 | 확장 과제 |

## 구현 순서 제안

1. `main`에서 `listenfd`, `accept` 루프 완성
2. 한 연결을 처리하는 함수 작성
3. 요청라인과 헤더 파싱
4. URI에서 `host`, `port`, `path` 추출
5. 서버용 요청 헤더 조립
6. 원격 서버 연결 후 요청 전달
7. 응답 relay
8. 순차 버전 검증
9. 스레드 도입
10. 캐시 도입

## 디버깅 관점의 체크포인트

- 요청라인이 기대한 형식인가
- `Host`를 제대로 추출했는가
- 포트를 생략한 URI를 `80`으로 처리하는가
- 빈 줄 `\r\n`로 헤더를 끝냈는가
- 응답 바이트를 중간에 끊지 않고 그대로 relay하는가
- 큰 응답을 캐시에 잘못 저장하지 않는가

## 공부하면서 함께 보면 좋은 개념

### HTTP/1.0 vs HTTP/1.1

- Tiny/Proxy Lab은 보통 단순화를 위해 `HTTP/1.0` 스타일에 맞춘다.
- 연결을 길게 유지하지 않고 한 번 요청 후 닫는 흐름이 디버깅하기 쉽다.

### REST API

- 프록시 과제는 REST 서버 구현이 목적은 아니다.
- 다만 HTTP 메서드와 헤더 구조를 읽는 감각은 REST API 디버깅에도 그대로 연결된다.

### CDN

- 캐시 프록시를 이해하면 CDN의 핵심 아이디어도 훨씬 쉽게 이해된다.
- 둘 다 중간 계층에서 응답을 대신 제공할 수 있기 때문이다.

## Proxy 시작 전 완료 조건

- Tiny가 동작한다.
- `Open_clientfd`와 `Open_listenfd`의 차이를 설명할 수 있다.
- HTTP 요청라인을 직접 문자열로 써볼 수 있다.
- `Host`, `path`, `port`를 URI에서 분리하는 로직을 말로 설명할 수 있다.
