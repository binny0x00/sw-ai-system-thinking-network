# 네트워크 프로그래밍을 위한 C 문법 빠른 가이드

이 문서는 C 언어가 익숙하지 않은 상태에서 `tiny.c`, `adder.c`, `proxy.c`를 읽고 수정하기 위해 필요한 문법만 추려서 정리한 것이다.

## 우선 익숙해져야 하는 문법 8개

### 1. 포인터

```c
int x = 10;
int *p = &x;
printf("%d\n", *p);
```

- `&x`: x의 주소
- `p`: 주소를 저장하는 포인터
- `*p`: 그 주소가 가리키는 실제 값

네트워크 코드에서는 함수가 구조체나 버퍼를 수정해야 해서 포인터를 자주 쓴다.

### 2. 문자열은 `char` 배열

```c
char buf[1024];
strcpy(buf, "GET / HTTP/1.0\r\n");
```

- C 문자열은 끝에 `\0`이 붙는다.
- 버퍼 크기를 넘기지 않도록 항상 주의해야 한다.

### 3. 배열과 포인터의 가까운 관계

```c
char buf[1024];
char *p = buf;
```

- 배열 이름은 자주 포인터처럼 동작한다.
- 그래서 `Rio_readlineb(&rio, buf, MAXLINE);` 같은 코드가 자연스럽다.

### 4. 구조체

```c
struct sockaddr_storage clientaddr;
socklen_t clientlen = sizeof(clientaddr);
```

- 여러 필드를 묶는 사용자 정의 타입이다.
- 소켓 주소는 구조체로 다룬다.

### 5. 함수 인자 전달

```c
void read_requesthdrs(rio_t *rp);
```

- `rio_t *rp`는 `rio_t` 구조체의 주소를 받는다는 뜻이다.
- 큰 구조체를 복사하지 않고 원본을 다루기 위해 포인터를 넘긴다.

### 6. `argc`, `argv`

```c
int main(int argc, char **argv)
```

- `argc`: 인자 개수
- `argv`: 인자 문자열 배열
- `argv[1]`은 보통 포트 번호 문자열이다.

### 7. 시스템 콜과 에러 처리

```c
listenfd = Open_listenfd(argv[1]);
```

- CSAPP 래퍼 함수는 실패 시 에러를 출력하고 종료하는 경우가 많다.
- 처음에는 이 래퍼를 믿고 흐름을 이해하는 데 집중하는 편이 좋다.

### 8. 프로세스 관련 함수

```c
if (Fork() == 0) {
    Execve(filename, emptylist, environ);
}
Wait(NULL);
```

- 동적 컨텐츠를 위해 Tiny는 CGI 프로그램을 별도 프로세스로 실행한다.
- 즉, `serve_dynamic`을 이해하려면 `fork`, `execve`, `wait` 감각이 필요하다.

## 코드 읽을 때 자주 보는 표현

### 문자열 비교

```c
if (strcasecmp(method, "GET")) {
    ...
}
```

- `strcasecmp(a, b)`가 `0`이면 두 문자열이 같다.
- 그래서 이 코드는 "GET이 아니면"으로 읽어야 한다.

### 포맷 문자열

```c
sprintf(buf, "HTTP/1.0 200 OK\r\n");
```

- 버퍼에 문자열을 써 넣는다.
- 나중에는 `snprintf`가 더 안전하다는 점도 알아두면 좋다.

### 문자열 찾기

```c
p = strchr(buf, '&');
```

- 문자열에서 특정 문자의 첫 위치를 찾는다.
- `adder.c`가 쿼리 문자열을 파싱할 때 사용한다.

### 조건 분기

```c
if (S_ISREG(sbuf.st_mode) && (S_IRUSR & sbuf.st_mode))
```

- 파일이 일반 파일인지, 읽기 권한이 있는지 검사한다.
- 초보자에게는 문법보다 "정적 파일을 안전하게 제공할 수 있는가"를 검사하는 코드라고 이해하는 편이 먼저다.

## `tiny.c`에서 꼭 읽어야 할 C 포인트

### `rio_t`

- CSAPP의 Robust I/O 추상화다.
- 줄 단위 읽기, 지정 길이 쓰기를 안전하게 처리하려고 만든 래퍼다.
- 네트워크에서 `read`/`write`를 그대로 쓰는 것보다 초보자에게 훨씬 안정적이다.

### `stat`

- 파일 정보 조회 함수다.
- Tiny는 요청한 파일이 실제로 존재하는지, 권한이 있는지 확인할 때 사용한다.

### `mmap`

- 파일 내용을 메모리에 매핑한다.
- Tiny는 정적 파일 응답을 빠르게 보내기 위해 이 기법을 사용한다.
- 처음에는 "파일을 읽어서 메모리에 올린 뒤 응답으로 보낸다" 수준으로 이해해도 충분하다.

### `dup2`

- 표준출력 같은 파일 디스크립터를 다른 대상으로 연결할 때 사용한다.
- CGI 프로그램의 출력이 곧 HTTP 응답 본문이 되도록 만들 때 중요하다.

## 초보자가 자주 하는 실수

- 문자열 버퍼 크기를 고려하지 않고 `strcpy`, `sprintf`를 남발한다.
- `==`로 문자열 비교를 시도한다.
- 포인터와 실제 값을 혼동한다.
- `listenfd`와 `connfd`를 같은 변수처럼 다룬다.
- 함수 선언만 보고 "왜 포인터를 받지?"에서 멈춘다.

## 추천 읽기 순서

1. `main`
2. `doit`
3. `parse_uri`
4. `serve_static`
5. `serve_dynamic`
6. `adder.c`

이 순서가 좋은 이유:

- 먼저 전체 요청 흐름을 잡고
- 그 다음 URI 해석
- 그 다음 정적/동적 응답 차이
- 마지막으로 CGI 프로그램을 별도 파일에서 확인할 수 있기 때문이다.

## 직접 손으로 써보면 좋은 연습

1. 문자열 `"GET /home.html HTTP/1.0\r\n"`를 세 변수 `method`, `uri`, `version`으로 나눠 저장해보기
2. `char buf[1024]`에 응답 헤더를 `sprintf`로 한 줄씩 붙여보기
3. 파일 경로를 받아 확장자에 따라 MIME 타입을 반환하는 함수 흉내 내보기

## 디버깅 팁

- 포인터가 헷갈리면 변수 주소와 값을 각각 출력해본다.
- 문자열 파싱이 꼬이면 중간 버퍼를 계속 `printf`해본다.
- 네트워크 흐름이 안 보이면 먼저 `printf("method=%s uri=%s\n", method, uri);` 같이 로그를 넣는다.
- 한 번에 다 이해하려 하지 말고 "이 함수의 입력은 무엇이고 출력은 무엇인가?"만 본다.
