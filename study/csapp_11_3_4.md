# 11.3 ~ 11.4 핵심 정리

이 문서는 `IP 인터넷`과 `BSD 소켓 인터페이스`를 과제 구현 관점에서 빠르게 이해하기 위한 요약이다.

## 11.3 Global IP Internet

### 왜 이 절이 중요한가

브라우저에서 URL을 입력하면 실제로는 "어느 호스트의 어느 프로세스에게 TCP 연결을 열 것인가?"라는 문제를 풀어야 한다.  
이 절은 그 답을 이루는 최소 단위인 `IP 주소`, `도메인 이름`, `포트`, `연결`을 설명한다.

### 1. IP 주소

- IP 주소는 네트워크 상의 호스트를 식별하는 주소다.
- 사람은 `www.example.com` 같은 이름을 기억하고, 컴퓨터는 IP 주소로 통신한다.
- 이번 과제에서는 숫자 자체를 외우는 것이 아니라 "`도메인 -> DNS -> IP`" 흐름을 이해하면 충분하다.

### 2. 도메인 이름과 DNS

- DNS는 도메인 이름을 IP 주소로 바꾸는 분산 데이터베이스 시스템이다.
- `open_clientfd(host, port)` 같은 함수 내부에서는 보통 `getaddrinfo` 계열 호출을 이용해 이름을 주소로 바꾼다.
- 즉, `Host` 헤더나 URL의 호스트 부분은 결국 소켓 연결을 만들기 위한 주소 정보로 변환된다.

### 3. 포트 번호

- 하나의 컴퓨터에 서버 프로세스는 여러 개 있을 수 있다.
- 그래서 "어느 컴퓨터인가?"만으로는 부족하고, "그 컴퓨터 안의 어느 서비스인가?"를 구분해야 한다.
- 이때 쓰는 번호가 `port`다.
- 예:
  - `80`: 일반적인 HTTP
  - `443`: HTTPS
  - `8000`, `8080`: 개발용 서버에서 자주 사용

### 4. 인터넷 연결

- 연결은 `(client IP, client port, server IP, server port)`의 조합으로 구분할 수 있다.
- 서버는 `listen socket`으로 새 연결을 기다리고, 실제 데이터 송수신은 `connected socket`으로 처리한다.
- Tiny 서버에서 `listenfd`와 `connfd`를 분리하는 이유가 바로 이것이다.

## 11.4 Sockets Interface

### 소켓을 한 줄로 설명하면

운영체제가 네트워크 통신을 파일처럼 다루게 해 주는 인터페이스다.

### file descriptor가 중요한 이유

- 유닉스에서는 파일, 파이프, 소켓을 모두 `file descriptor(fd)`로 다룬다.
- 그래서 `read`, `write`, `close` 같은 사고방식이 네트워크에도 이어진다.
- Tiny 코드에서 `connfd`가 그냥 정수 하나인 이유도 이것 때문이다.

## 서버 쪽 흐름

```text
socket
  -> bind
  -> listen
  -> accept
  -> read/write
  -> close
```

### `socket`

- "통신 끝점(endpoint)을 하나 만들겠다"는 선언이다.
- 아직 누구와 통신할지는 정해지지 않았다.

### `bind`

- 만든 소켓에 로컬 주소와 포트를 붙인다.
- 서버는 보통 특정 포트에서 요청을 기다려야 하므로 `bind`가 필요하다.

### `listen`

- 이 소켓을 "연결 요청을 받는 소켓"으로 전환한다.
- 즉, 읽고 쓰는 일반 소켓이 아니라 "대기용 소켓"이 된다.

### `accept`

- 클라이언트가 접속하면 새 연결 전용 소켓을 만들어 반환한다.
- 기존 `listenfd`는 계속 새 클라이언트를 받기 위해 남겨둔다.
- 반환된 `connfd`로 HTTP 요청을 읽고 응답을 보낸다.

## 클라이언트 쪽 흐름

```text
socket
  -> connect
  -> read/write
  -> close
```

### `connect`

- 서버 주소와 포트로 연결을 시도한다.
- 브라우저나 프록시는 이 흐름을 사용한다.
- Tiny는 서버이므로 `listen/accept`를 쓰고, Proxy는 서버이면서 동시에 웹 서버로 연결하는 클라이언트 역할도 한다.

## Tiny와 Proxy를 연결해서 보기

| 프로그램 | 역할 | 사용하는 핵심 흐름 |
| --- | --- | --- |
| `tiny` | 서버 | `Open_listenfd -> Accept -> Rio_readlineb -> Rio_writen` |
| 브라우저 | 클라이언트 | `connect -> send request -> read response` |
| `proxy` | 둘 다 | 브라우저에게는 서버, 원격 웹서버에게는 클라이언트 |

## 초보자용 비유

- `listen socket`: 가게의 대표 번호
- `accept` 후 `connfd`: 손님 한 명과 통화 중인 개별 전화선
- `HTTP request`: 손님 주문서
- `HTTP response`: 가게가 주는 결과물

## 과제에서 자주 헷갈리는 포인트

### 1. `listenfd`와 `connfd`를 혼동함

- `listenfd`는 새 연결을 기다리는 용도다.
- 실제 HTTP 메시지는 `connfd`로 읽고 쓴다.

### 2. `fd`가 파일 전용이라고 생각함

- 소켓도 `fd`다.
- 그래서 "소켓은 파일처럼 읽고 쓸 수 있다"는 감각이 중요하다.

### 3. `DNS`, `IP`, `Port`의 역할을 섞어 생각함

- `DNS`: 이름을 주소로 바꿈
- `IP`: 어느 호스트인가
- `Port`: 그 호스트 안의 어느 서비스인가

## 이번 과제에 필요한 만큼만 보는 OSI 7계층

모든 계층을 깊게 파지 않아도 된다. 아래 위치 관계만 이해하면 충분하다.

| 계층 관점 | 이번 과제에서 보는 것 |
| --- | --- |
| 응용 계층 | `HTTP`, `REST API` |
| 전송 계층 | `TCP`, `UDP` |
| 인터넷 계층 | `IP` |

- `HTTP`는 사람이 읽을 수 있는 요청/응답 규칙이다.
- `TCP`는 그 HTTP 문자열을 순서대로 안전하게 전달한다.
- `IP`는 패킷이 어느 호스트로 갈지 결정한다.

## REST API와 HTTP 메서드

이번 과제의 Tiny는 본격적인 REST 서버는 아니지만, 웹 프로그래밍 감각을 익히려면 아래 정도는 알고 가는 편이 좋다.

- `GET`: 조회
- `POST`: 생성/처리 요청
- `PUT` 또는 `PATCH`: 수정
- `DELETE`: 삭제

주의할 점:

- 사용자 요청에 적힌 `UPDATE`는 HTTP 표준 메서드 이름이 아니다.
- 보통 수정은 `PUT` 또는 `PATCH`로 표현한다.
- Tiny 구현은 우선 `GET` 처리에 집중한다.

## CDN과 Proxy의 연결

- `CDN`은 사용자와 가까운 곳에서 정적 컨텐츠를 캐시해 빠르게 제공하는 인프라다.
- `Proxy`는 요청을 대신 전달하는 중간 서버다.
- 둘 다 "중간 계층에서 요청/응답을 다룬다"는 공통점이 있어, 프록시 과제를 이해할 때 CDN 개념이 도움이 된다.
